Understanding CodeConcise

Before modernizing a legacy codebase using Twist, we need comprehensive Business Requirements Documents (BRDs) for that legacy code. To create these BRDs, we first need to understand the legacy code at a deep level. That's where CodeConcise comes in as a powerful tool.

What is CodeConcise?

CodeConcise is a Graph RAG (Retrieval-Augmented Generation) platform that:

  • 1. Ingests Legacy Code: Takes a complete codebase as input
  • 2. Builds Knowledge Graph: Creates a semantic graph of all code relationships, dependencies, and patterns
  • 3. Enriches with Metadata: Uses LLM to add business context, purpose, and relationships to each code element
  • 4. Exposes APIs: Provides intelligent query interfaces to extract information

Why This Matters

Traditional code analysis tools only understand syntax. CodeConcise understands:

  • ✓ Business Purpose: What each component does for the business
  • ✓ Relationships: How components interact and depend on each other
  • ✓ Patterns: Recurring architectural and design patterns
  • ✓ Capabilities: High-level business capabilities the system provides

Deep dive into the 6-phase workflow (optional)

Detailed Phase Breakdown

1

INITIALIZE: Load Capabilities from CodeConcise

What happens: Agent calls CodeConcise Capability Structure API to fetch all capabilities

Example: Fetches 56 capabilities from retail-banking repository

2

SELECT CAPABILITY: Pick Next Capability to Document

What happens: Agent selects the next unprocessed capability from the list

Example: Selects "User Authentication" as the first capability to document

3

MODEL: Claude LLM Plans & Decides Next Action

What happens: Claude LLM analyzes the selected capability and decides what information is needed for ALL 5 document types

Decision Logic: LLM evaluates current context and decides:

→ Need more information? Call codeconcise_api tool with targeted queries
→ Have enough context? Call document_writer tool 5 times to generate all documents

Key Insight: LLM uses contextual understanding from previous responses to formulate increasingly specific follow-up queries

4

TOOLS: Execute Queries & Write Documents ⭐

What happens: LLM executes tools based on its decision from MODEL phase

🔄 Information Gathering Loop (Iterative):

• Query 1: "What is the business purpose and stakeholders?" (minimal params)
• Query 2: Based on response 1, ask follow-up: "What are the key workflows?" (escalate if needed)
• Query 3: Based on responses 1-2, ask: "What are the technical details?" (escalate if needed)
• Query 4: Based on responses 1-3, ask: "What are risks and improvements?" (escalate if needed)
• Query 5: Based on all previous, ask: "What is the architecture?" (escalate if needed)

Each query is contextually informed by previous responses

📝 Document Generation (When Ready):

• Call document_writer 5 times to write:
1. functional_summary.md
2. functional_flow.md
3. functional_summary.md
4. insightful_summary.md
5. architecture.md

All documents use complete accumulated context

Information Accumulation: All query responses stored in message history, building comprehensive knowledge base

Intelligent Escalation: If responses insufficient, LLM increases top_k and neighbours_depth for deeper analysis

⏱️ Total time: ~3 minutes per capability

5

MODEL → TOOLS Loop Repeats Until Ready

What happens: After each CodeConcise query, control returns to MODEL phase

LLM Decision: Evaluates accumulated context and decides:

❓ Need more information? → Loop back to TOOLS for more queries
✅ Have enough context? → Proceed to document generation

Contextual Understanding: Each iteration builds on previous responses, making queries more specific and targeted

Result: Comprehensive knowledge base accumulated in message history

6

QA COVERAGE GATE: Validate & Optimize

What happens: Agent verifies all 5 documents were generated successfully and meet quality criteria

Quality Checks:

✓ All 5 documents exist
✓ Each meets minimum word count
✓ Content requirements satisfied

If Successful: Clears message history after capability completion to prevent context bloat (key to 29.6% performance improvement!)

If Failed: Escalates query parameters and retries document generation, resulting in 99.5% success rate (555/556 documents)

🔄 The Complete Flow:

1. SELECT CAPABILITY → 2. MODEL (plan info gathering)
3. TOOLS (query CodeConcise) → 4. MODEL (decide: more queries or write docs?)
5. If more info needed: Loop back to TOOLS
6. If ready: TOOLS (call document_writer 5 times)
7. QA COVERAGE GATE (validate & clear history)
8. Repeat for next capability

Final Result: 5 structured documents per capability (e.g., 56 capabilities × 5 documents = 280 total documents)

Total Time: ~3.5 hours for all 56 capabilities (57% faster than initial implementation)
